1//! Wrappers around the [CUDA driver API](https://docs.nvidia.com/cuda/cuda-driver-api/index.html),
2//! in three levels. See crate documentation for description of each.
3//!
4//! # safe api usage
5//!
6//! 1. Instantiate a [CudaDevice]:
7//!
8//! ```rust
9//! # use cudarc::driver::*;
10//! let device = CudaDevice::new(0).unwrap();
11//! ```
12//!
13//! 2. Allocate device memory with host data with [CudaDevice::htod_copy()], [CudaDevice::alloc_zeros()],
14//!    or [CudaDevice::htod_sync_copy()].
15//!
16//! You can also copy data to CudaSlice using [CudaDevice::htod_sync_copy_into()]
17//!
18//! ```rust
19//! # use cudarc::driver::*;
20//! # let device = CudaDevice::new(0).unwrap();
21//! let a_dev: CudaSlice<f32> = device.alloc_zeros(10).unwrap();
22//! let b_dev: CudaSlice<f32> = device.htod_copy(vec![0.0; 10]).unwrap();
23//! let c_dev: CudaSlice<f32> = device.htod_sync_copy(&[1.0, 2.0, 3.0]).unwrap();
24//! ```
25//!
26//! 3. Transfer to host memory with [CudaDevice::sync_reclaim()], [CudaDevice::dtoh_sync_copy()],
27//!    or [CudaDevice::dtoh_sync_copy_into()]
28//!
29//! ```rust
30//! # use cudarc::driver::*;
31//! # use std::rc::Rc;
32//! # let device = CudaDevice::new(0).unwrap();
33//! let a_dev: CudaSlice<f32> = device.alloc_zeros(10).unwrap();
34//! let mut a_buf: [f32; 10] = [1.0; 10];
35//! device.dtoh_sync_copy_into(&a_dev, &mut a_buf);
36//! assert_eq!(a_buf, [0.0; 10]);
37//! let a_host: Vec<f32> = device.sync_reclaim(a_dev).unwrap();
38//! assert_eq!(&a_host, &[0.0; 10]);
39//! ```
40//!
41//! ## Mutating device memory - [CudaFunction]
42//!
43//! See [LaunchAsync] and [CudaFunction].
44//!
45//! In order to mutate device data, you need to use cuda kernels.
46//!
47//! Loading kernels is done with [CudaDevice::load_ptx()]
48//! ```rust
49//! # use cudarc::{driver::*, nvrtc::*};
50//! let ptx = compile_ptx("extern \"C\" __global__ void my_function(float *out) { }").unwrap();
51//! let device = CudaDevice::new(0).unwrap();
52//! device.load_ptx(ptx, "module_name", &["my_function"]).unwrap();
53//! ```
54//!
55//! Retrieve the function using the registered module name & actual function name:
56//! ```rust
57//! # use cudarc::{driver::*, nvrtc::*};
58//! # let ptx = compile_ptx("extern \"C\" __global__ void my_function(float *out) { }").unwrap();
59//! # let device = CudaDevice::new(0).unwrap();
60//! # device.load_ptx(ptx, "module_name", &["my_function"]).unwrap();
61//! let func: CudaFunction = device.get_func("module_name", "my_function").unwrap();
62//! ```
63//!
64//! Asynchronously execute the kernel:
65//! ```rust
66//! # use cudarc::{driver::*, nvrtc::*};
67//! # let ptx = compile_ptx("extern \"C\" __global__ void my_function(float *out) { }").unwrap();
68//! # let device = CudaDevice::new(0).unwrap();
69//! # device.load_ptx(ptx, "module_name", &["my_function"]).unwrap();
70//! # let func: CudaFunction = device.get_func("module_name", "my_function").unwrap();
71//! let mut a = device.alloc_zeros::<f32>(10).unwrap();
72//! let cfg = LaunchConfig::for_num_elems(10);
73//! unsafe { func.launch(cfg, (&mut a,)) }.unwrap();
74//! ```
75//!
76//! Note: Launching kernels is **extremely unsafe**. See [LaunchAsync] for more info.
77//!
78//! ## Sub slices of [CudaSlice]
79//!
80//! For some operations, it is necessary to only operate on a small part of a single [CudaSlice].
81//! For example, the slice may represent a batch of items, and you want to run separate kernels
82//! on each of the items in the batch.
83//!
84//! Use [CudaSlice::try_slice()] and [CudaSlice::try_slice_mut()] for this. The returned
85//! views ([CudaView] and [CudaViewMut] hold references to the owning [CudaSlice],
86//! so rust's ownership system handles safety here.
87//!
88//! These view structs can be used with [CudaFunction].
89//!
90//! ```rust
91//! # use cudarc::{driver::*, nvrtc::*};
92//! # let ptx = compile_ptx("extern \"C\" __global__ void my_function(float *out) { }").unwrap();
93//! # let device = CudaDevice::new(0).unwrap();
94//! # device.load_ptx(ptx, "module_name", &["my_function"]).unwrap();
95//! let mut a: CudaSlice<f32> = device.alloc_zeros::<f32>(3 * 10).unwrap();
96//! for i_batch in 0..3 {
97//!     let mut a_sub_view: CudaViewMut<f32> = a.try_slice_mut(i_batch * 10..).unwrap();
98//!     let f: CudaFunction = device.get_func("module_name", "my_function").unwrap();
99//!     let cfg = LaunchConfig::for_num_elems(10);
100//!     unsafe { f.launch(cfg, (&mut a_sub_view,)) }.unwrap();
101//! }
102//! ```
103//!
104//! #### A note on implementation
105//!
106//! It would be possible to re-use [CudaSlice] itself for sub-slices, however that would involve adding
107//! another structure underneath the hood that is wrapped in an [std::sync::Arc] to minimize data cloning.
108//! Overall it seemed more complex than the current implementation.
109//!
110//! # Multi threading
111//!
112//! In order to use a [CudaDevice] on multiple threads, you must call [CudaDevice::bind_to_thread] on
113//! each thread **before you use the device**.
114//!
115//! # Safety
116//!
117//! There are a number of aspects to this, but at a high level this API utilizes [std::sync::Arc] to
118//! control when [CudaDevice] can be dropped.
119//!
120//! ### Context/Stream lifetimes
121//!
122//! The first part of safety is ensuring that [crate::driver::sys::CUcontext],
123//! [crate::driver::sys::CUdevice], and [crate::driver::sys::CUstream] all
124//! live the required amount of time (i.e. device outlives context, which outlives stream).
125//!
126//! This is accomplished by putting all of them inside one struct, the [CudaDevice]. There are other ways,
127//! such as adding newtypes that carry lifetimes with them, but this approach was chosen to make working
128//! with device pointers easier.
129//!
130//! Additionally, [CudaDevice] implements [Drop] as releasing all the data from the device in
131//! the expected way.
132//!
133//! ### Device Data lifetimes
134//!
135//! The next part of safety is ensuring that [CudaSlice] do not outlive
136//! the [CudaDevice]. For usability, each [CudaSlice] owns an `Arc<CudaDevice>`
137//! to ensure the device stays alive.
138//!
139//! Additionally we don't want to double free any device pointers, so free is only
140//! called when the device pointer is dropped. Thanks rust!
141//!
142//! ### Host and Device Data lifetimes
143//!
144//! Each device allocation can be associated with a host allocation. We want to ensure
145//! that these have the same lifetimes *when copying data between them*.
146//!
147//! This is done via the various copy methods. Methods that don't take ownership
148//! of the host data need to be executed synchronously, while the methods own the reference.
149//! Methods that do own the host data can be executed synchronously.
150//!
151//! ### Single stream operations
152//!
153//! The next part of safety is ensuring that all operations happen on a single stream.
154//! This ensures that data isn't mutated by more than 1 stream at a time, and also
155//! ensures data isn't used before allocated, or used after free.
156//!
157//! Another important aspect of this is ensuring that mutability in an async setting
158//! is sound, and something can't be freed while it's being used in a kernel.
159//!
160//! To this end every operation by default happens on the same stream.
161//!
162//! Multi stream is supported via [CudaStream], however it automatically
163//! synchronizes with the main stream on creation & on drop. It is still possible
164//! to be unsafe in a multi stream context though.
